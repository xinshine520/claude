## 1. 概述

### 1.1 产品愿景

构建一个基于 Python 的 MCP (Model Context Protocol) 服务器，使 MCP 客户端（如 Cursor、Claude Desktop 等）能够通过自然语言描述来查询 PostgreSQL 数据库。服务器利用 DeepSeek 大模型（deepseek-chat）将自然语言转换为 SQL，并在执行前进行安全校验与结果验证，最终向用户返回生成的 SQL 语句或查询结果。

### 1.2 目标用户

- 使用 MCP 兼容客户端的开发者、数据分析师和数据库管理员
- 需要以自然语言方式快速查询 PostgreSQL 数据库的技术人员

### 1.3 核心价值

- 降低 SQL 编写门槛，通过自然语言即可完成数据库查询
- 自动感知数据库结构，无需用户手动描述 schema
- 内置安全防护，仅允许只读查询，防止数据篡改
- 智能结果验证，确保返回有意义的查询结果

## 2. 功能需求

### 2.1 数据库 Schema 自动发现与缓存

#### 2.1.1 启动时 Schema 采集

- **FR-2.1.1-01**: 服务器启动时，自动连接到所有已配置的 PostgreSQL 数据库实例，验证连接可用性并采集 schema 摘要（数据库名、schema 列表、表/视图数量）
- **FR-2.1.1-02**: 详细 schema 信息采用**懒加载**策略：首次查询某数据库时才拉取完整元数据，避免启动时间过长
- **FR-2.1.1-03**: 对每个数据库，完整采集包含以下元数据（缓存到内存）：
  - **Tables（表）**: 表名、列名、列类型、列约束（NOT NULL / PRIMARY KEY / UNIQUE / DEFAULT 等）、列注释（如有）
  - **Views（视图）**: 视图名、列定义；视图底层查询定义为可选采集项（可通过配置开关控制，默认采集）
  - **Types（自定义类型）**: 枚举类型、复合类型等用户自定义数据类型的名称与定义
  - **Indexes（索引）**: 索引名、所属表、索引列、索引类型（B-tree / Hash / GIN / GiST 等）、是否唯一
  - **Schemas（模式）**: 所有非系统 schema 的名称（排除 `pg_catalog`、`information_schema` 等）
  - **Foreign Keys（外键关系）**: 外键约束名称、来源表/列、目标表/列，用于理解表间关联
  - **表/列注释（Comments）**: 通过 `pg_description` 获取表和列的注释信息，作为语义上下文提供给大模型
- **FR-2.1.1-04**: Schema 采集过程应记录日志，包括每个数据库的采集结果摘要（表数量、视图数量等）
- **FR-2.1.1-05**: 如果某个数据库连接失败，应记录错误日志并继续处理其他数据库，不应导致服务器整体启动失败
- **FR-2.1.1-06**: 如果连接用户缺少部分元数据的查询权限（如视图定义、注释），应优雅降级（记录警告、跳过缺失字段），不阻塞整体采集

#### 2.1.2 Schema 缓存管理

- **FR-2.1.2-01**: 缓存的 Schema 信息应存储在内存中，支持按数据库名和 schema 名快速检索
- **FR-2.1.2-02**: 提供 Schema 缓存刷新机制，允许在数据库结构变更后更新缓存（用户可通过自然语言查询触发，如"刷新 schema 缓存"）
- **FR-2.1.2-03**: 刷新操作应支持按指定数据库刷新，也支持全量刷新（通过自然语言指令中的数据库名称识别）

### 2.2 自然语言转 SQL

#### 2.2.1 用户输入

- **FR-2.2.1-01**: 用户通过 MCP tool 调用提交自然语言查询请求，请求中包含：
  - **查询描述（必填）**: 自然语言描述的查询需求
  - **目标数据库（可选）**: 指定要查询的数据库名称；若未指定，服务器根据查询内容和已缓存的 schema 信息自动推断最合适的数据库
  - **返回模式（可选）**: 指定返回 SQL 语句还是查询结果，默认返回查询结果
    - `sql` — 仅返回生成的 SQL 语句
    - `result` — 执行 SQL 并返回查询结果
  - **结果行数限制（可选）**: 限制返回结果的最大行数，默认为 100 行

#### 2.2.2 SQL 生成

- **FR-2.2.2-01**: 服务器将用户的自然语言描述与目标数据库的 schema 信息组合为 prompt，调用 DeepSeek deepseek-chat 模型生成 SQL
- **FR-2.2.2-02**: 采用**检索增强**策略构建上下文，避免发送完整 schema（可能超出上下文窗口或成本过高）：
  - 先根据用户查询中的关键词/实体，在缓存中检索相关表和列（按表名、列名、注释匹配）
  - 仅将相关子集 schema（表结构、关系、注释）发送给模型
  - 对视图定义和注释设置长度上限（可配置，建议 500 字符），超出部分截断
  - 设置 token 预算上限，当 schema 子集仍超出预算时进一步裁剪
- **FR-2.2.2-03**: 发送给大模型的 prompt 应使用**固定模板**，明确分离系统指令与用户内容，防止 prompt 注入：
  - 系统指令部分：只生成 SELECT/只读查询语句、输出格式要求（仅输出可直接执行的 SQL）
  - 上下文部分：目标数据库的相关 schema 信息
  - 用户输入部分：自然语言查询描述（作为引用内容，不直接拼接到指令中）
- **FR-2.2.2-04**: 如果用户未指定目标数据库，系统应**优先在本地**进行匹配（基于关键词/表名在各数据库 schema 摘要中的命中率），仅在本地无法确定时才调用大模型辅助判断，且只发送最小必要的摘要信息（数据库名 + 表名列表）
- **FR-2.2.2-05**: 核心安全原则 — **LLM 输出一律不可信**，生成的 SQL 必须经过独立的代码层安全校验（§2.3），不依赖 prompt 中的"只读"指令来保证安全

### 2.3 SQL 安全校验

- **FR-2.3-01**: 使用 SQL 解析器（如 `pglast` / `sqlglot`）对生成的 SQL 进行 **AST 级别**解析，确保其为合法的 PostgreSQL 语句
- **FR-2.3-02**: **必须为单语句**：拒绝包含分号分隔的多语句（stacked queries），解析结果必须为恰好一个语句节点
- **FR-2.3-03**: 仅允许以下 AST 节点类型通过校验（严格白名单）：
  - `SelectStmt`（含子查询、UNION/INTERSECT/EXCEPT）
  - `CommonTableExpr`（CTE，但 CTE 体必须也是 `SelectStmt`）
  - `ExplainStmt`（仅 `EXPLAIN`，**默认禁止 `EXPLAIN ANALYZE`**，因为 ANALYZE 会实际执行语句并可能获取锁；可通过配置开关允许）
- **FR-2.3-04**: 必须拒绝以下内容（黑名单 + 白名单互补）：
  - 数据修改语句：`INSERT`、`UPDATE`、`DELETE`、`TRUNCATE`、`MERGE`
  - 结构修改语句：`CREATE`、`ALTER`、`DROP`（包括 `CREATE TEMP TABLE AS SELECT`）
  - 权限管理语句：`GRANT`、`REVOKE`
  - 过程/事务控制：`CALL`、`DO`、`PREPARE`、`EXECUTE`、`DEALLOCATE`、`SET`、`LISTEN`、`NOTIFY`、`COPY`
  - 包含 `INTO` 子句的 SELECT 语句（`SELECT ... INTO`）
- **FR-2.3-05**: **危险函数黑名单** — 在 AST 中检测到以下函数调用时必须拒绝：
  - 系统管理：`pg_sleep`、`pg_terminate_backend`、`pg_cancel_backend`、`pg_reload_conf`
  - 文件/网络访问：`pg_read_file`、`pg_write_file`、`pg_read_binary_file`、`lo_export`、`lo_import`
  - 危险操作：`set_config`、`current_setting`（写入模式）、`dblink`、`dblink_exec`
  - 可配置的自定义黑名单（允许运维人员按需扩展）
- **FR-2.3-06**: 对超长 SQL（超过可配置阈值，建议 10,000 字符）直接拒绝，防止资源消耗型攻击
- **FR-2.3-07**: 如果 SQL 未通过安全校验，返回结构化的错误信息（包含拒绝原因和违规类型），**不返回生成的 SQL 原文**（防止用户绕过系统手动执行危险查询）

### 2.4 SQL 执行与结果获取

- **FR-2.4-01**: 通过安全校验的 SQL 应在目标数据库上执行
- **FR-2.4-02**: 执行时必须使用以下事务模板，确保只读与超时保护：
  ```sql
  BEGIN READ ONLY;
  SET LOCAL statement_timeout = '30s';
  SET LOCAL lock_timeout = '5s';
  -- 用户 SQL
  COMMIT;
  ```
  其中 `statement_timeout` 和 `lock_timeout` 值均可配置
- **FR-2.4-03**: 必须使用**连接池**管理数据库连接，配置包括：
  - 每数据库最大连接数（建议默认 5）
  - 连接获取超时时间（建议默认 10 秒）
  - 全局最大并发查询数限制
  - 空闲连接回收策略
- **FR-2.4-04**: 获取查询结果时应遵循多维度限制，避免返回海量数据：
  - 行数限制：用户指定或默认值（100 行）
  - 单个字段值大小限制：超过阈值（可配置，建议 10KB）的字段值进行截断并标注
  - 总响应 payload 大小限制（可配置，建议 5MB）
- **FR-2.4-05**: 如果 SQL 执行失败，应捕获数据库错误并返回**脱敏后的**用户友好错误摘要；完整错误信息仅在 DEBUG 日志级别记录到服务器日志

### 2.5 结果语义验证

- **FR-2.5-01**: 语义验证为可选步骤，用户可通过参数控制是否启用，**默认关闭**（因涉及向外部 LLM 发送查询结果数据，存在数据安全风险）
- **FR-2.5-02**: 语义验证支持两种模式（可配置）：
  - **元数据模式（默认）**: 仅向 LLM 发送用户查询、生成的 SQL、结果的元数据（列名、列类型、行数、基本统计），**不发送实际数据行**
  - **采样模式**: 向 LLM 发送结果的前 N 行样本数据（N 可配置，建议 5 行）；启用此模式需用户显式确认（opt-in），并受数据治理策略约束
- **FR-2.5-03**: 采样模式的数据治理策略：
  - 支持列级 allowlist/denylist 配置，可排除敏感列（如邮箱、手机号、身份证号等）
  - 对发送的样本数据进行脱敏处理（可配置掩码规则）
  - 单次发送的样本数据量不超过可配置上限
- **FR-2.5-04**: 大模型应评估并返回：
  - 结果是否与用户意图匹配（是/否/部分匹配）
  - 如果不匹配或部分匹配，给出原因说明
  - 如果有改进建议，提供优化后的 SQL（可选）
- **FR-2.5-05**: 如果语义验证认为结果不匹配且有优化建议，系统可选择自动重试（最大重试次数 2 次；重试的 SQL 同样必须经过 §2.3 安全校验）
- **FR-2.5-06**: 最终返回给用户的响应应包含验证结果的摘要信息（如匹配度评价）

### 2.6 结果返回

- **FR-2.6-01**: 根据用户指定的返回模式返回相应内容：
  - **`sql` 模式**: 返回生成的 SQL 语句，以及语义验证摘要（如果启用了验证）
  - **`result` 模式**: 返回查询结果数据，同时附上生成的 SQL 和语义验证摘要
- **FR-2.6-02**: 查询结果应以结构化格式返回，包含：
  - `columns`: 列名及类型列表
  - `rows`: 数据行（列表形式）
  - `returned_row_count`: 实际返回的行数
  - `truncated`: 是否被截断（布尔值）
  - `total_row_count`: 符合条件的总行数（可选字段，可能为 null；不保证精确，不额外执行 COUNT 查询）
- **FR-2.6-03**: 对于大结果集，应在返回中明确标注结果已截断
- **FR-2.6-04**: 错误响应应使用统一的结构化格式：
  - `code`: 机器可读错误码（如 `VALIDATION_FAILED`、`EXECUTION_TIMEOUT`、`LLM_ERROR`）
  - `message`: 用户友好的错误描述
  - `stage`: 错误发生的阶段（`parsing`、`validation`、`execution`、`verification`）
  - `retryable`: 是否建议重试（布尔值）
  - 安全校验失败时**不返回**生成的 SQL 原文

## 3. MCP 接口定义

### 3.1 MCP Tool

服务器暴露唯一一个 MCP Tool：

#### `query` — 自然语言查询

主功能工具，接受自然语言并返回 SQL 或查询结果。

**输入参数:**

| 参数名          | 类型     | 必填 | 默认值     | 说明                                                  |
| --------------- | -------- | ---- | ---------- | ----------------------------------------------------- |
| `question`      | string   | 是   | —          | 自然语言描述的查询需求                                |
| `database`      | string   | 否   | 自动推断   | 目标数据库名称                                        |
| `return_mode`   | string   | 否   | `"result"` | 返回模式：`"sql"` 或 `"result"`                       |
| `max_rows`      | integer  | 否   | `100`      | 返回结果的最大行数（上限可配置）                      |
| `verify_result` | boolean  | 否   | `false`    | 是否启用语义验证（默认关闭，见 §2.5）                 |

**成功输出:**

| 字段名                | 类型     | 说明                                                  |
| --------------------- | -------- | ----------------------------------------------------- |
| `sql`                 | string   | 生成的 SQL 语句（安全校验失败时不返回）               |
| `database`            | string   | 实际查询的数据库名称                                  |
| `result`              | object   | 查询结果（仅 `result` 模式）                          |
| `result.columns`      | array    | 列定义列表，每项含 `name` 和 `type`                   |
| `result.rows`         | array    | 数据行                                                |
| `result.returned_row_count` | integer | 实际返回行数                                    |
| `result.truncated`    | boolean  | 结果是否被截断                                        |
| `result.total_row_count` | integer \| null | 总行数估算（可选，不保证精确）             |
| `verification`        | object   | 语义验证结果（仅启用验证时）；含 `match`、`explanation` 字段 |

**错误输出:**

| 字段名          | 类型     | 说明                                                  |
| --------------- | -------- | ----------------------------------------------------- |
| `error.code`    | string   | 机器可读错误码（如 `VALIDATION_FAILED`、`EXECUTION_TIMEOUT`、`LLM_ERROR`、`DB_UNAVAILABLE`） |
| `error.message` | string   | 用户友好的错误描述（脱敏后）                          |
| `error.stage`   | string   | 错误阶段：`parsing`、`validation`、`execution`、`verification` |
| `error.retryable` | boolean | 是否建议重试                                         |

> **说明:** 本期仅提供 `query` 一个 Tool，保持接口简洁。数据库列表、表结构查看、Schema 刷新等辅助功能均可通过 `query` 的自然语言能力间接实现（如用户可询问"列出所有表"或"描述 users 表的结构"）。后续版本可根据实际需求增加专用 Tool。

## 4. 非功能需求

### 4.1 安全性

#### 4.1.1 威胁模型与访问控制

- **NFR-4.1.1-01**: MCP 服务器默认仅监听本地（localhost / stdio），不暴露到网络；如需远程访问，必须配置 TLS
- **NFR-4.1.1-02**: 服务器支持可选的访问令牌（Bearer Token）认证；远程部署时必须启用
- **NFR-4.1.1-03**: 支持按数据库维度的访问控制配置（可限制特定客户端只能访问指定数据库）

#### 4.1.2 凭据与密钥管理

- **NFR-4.1.2-01**: 数据库连接凭据不得硬编码在代码中，必须通过环境变量或配置文件提供
- **NFR-4.1.2-02**: DeepSeek API Key 必须通过环境变量提供
- **NFR-4.1.2-03**: 敏感信息（连接字符串、API Key、查询结果中的数据）不得出现在日志中

#### 4.1.3 纵深防御

- **NFR-4.1.3-01**: 所有数据库操作必须使用只读连接/事务，从数据库层面防止意外写入
- **NFR-4.1.3-02**: SQL 安全校验必须在代码 AST 层面进行（不依赖 prompt 指令或数据库权限）
- **NFR-4.1.3-03**: LLM 输出一律视为不可信输入，必须经过完整的安全校验流水线

### 4.2 性能

- **NFR-4.2-01**: Schema 缓存应使服务器在接收到查询请求时不再需要实时查询数据库元数据（首次懒加载除外）
- **NFR-4.2-02**: 单次查询的端到端延迟预算（不含数据库查询执行时间）：
  - LLM SQL 生成：≤ 10 秒
  - 安全校验：≤ 500 毫秒
  - LLM 语义验证（如启用）：≤ 10 秒
  - 总计（不含重试）：≤ 15 秒；含重试时 ≤ 30 秒
- **NFR-4.2-03**: SQL 执行超时可配置，默认 30 秒
- **NFR-4.2-04**: 连接池配置：每数据库默认最大 5 连接，全局并发查询上限可配置

### 4.3 可靠性

- **NFR-4.3-01**: 单个数据库连接失败不应影响服务器对其他数据库的服务能力
- **NFR-4.3-02**: DeepSeek API 调用失败时应返回明确的错误信息，建议用户重试
- **NFR-4.3-03**: 数据库连接池应具备健康检查、断线重连与指数退避重试机制
- **NFR-4.3-04**: 当数据库持续不可用时，触发熔断机制，暂时标记该数据库为不可用，避免请求堆积

### 4.4 可观测性

- **NFR-4.4-01**: 记录结构化日志，包含每次查询的完整生命周期（请求 ID → 解析 → SQL 生成 → 安全校验 → 执行 → 验证 → 返回），每个阶段记录耗时
- **NFR-4.4-02**: 日志级别应可配置（DEBUG / INFO / WARNING / ERROR）
- **NFR-4.4-03**: **日志红线** — 以下内容在任何日志级别下均不得记录：
  - 数据库连接凭据、API Key
  - 查询结果的实际数据行
  - LLM prompt 中包含的用户数据
  - INFO 级别：仅记录 SQL 的 hash 摘要，不记录 SQL 全文（DEBUG 级别可记录 SQL 全文）

### 4.5 可配置性

- **NFR-4.5-01**: 以下参数应通过环境变量或配置文件进行配置：
  - PostgreSQL 数据库连接信息（支持多个数据库）
  - DeepSeek API Key、API Base URL、模型名称
  - SQL 执行超时时间、锁超时时间
  - 默认结果行数限制、单字段大小限制、总 payload 大小限制
  - 连接池大小、并发查询上限
  - 语义验证模式（关闭/元数据/采样）及采样行数
  - 危险函数黑名单（可扩展）
  - 日志级别
  - 访问令牌（可选）

## 5. 配置方式

### 5.1 数据库连接配置

通过环境变量配置一个或多个 PostgreSQL 数据库连接：

```
PG_MCP_DATABASES=db1,db2,db3
PG_MCP_DB1_HOST=localhost
PG_MCP_DB1_PORT=5432
PG_MCP_DB1_DATABASE=mydb1
PG_MCP_DB1_USER=readonly_user
PG_MCP_DB1_PASSWORD=secret
PG_MCP_DB1_SSLMODE=prefer
```

或使用连接字符串：


## 7. 错误场景与处理

> **原则:** 错误信息默认脱敏，仅返回用户友好摘要 + 错误码；完整错误详情仅记录到服务器日志（DEBUG 级别）。安全校验失败时不返回生成的 SQL 原文。

| 场景                           | 错误码                | 处理方式                                                   |
| ------------------------------ | --------------------- | ---------------------------------------------------------- |
| 数据库连接失败（启动时）       | —                     | 记录错误日志，跳过该数据库，继续启动；标记为不可用         |
| 数据库连接失败（查询时）       | `DB_UNAVAILABLE`      | 返回脱敏错误，提示用户该数据库暂不可用；`retryable: true`  |
| 数据库熔断触发                 | `DB_CIRCUIT_OPEN`     | 返回脱敏错误，提示数据库暂时不可用；`retryable: true`      |
| DeepSeek API 调用失败          | `LLM_ERROR`           | 返回脱敏错误，提示稍后重试；`retryable: true`              |
| DeepSeek 返回非法内容          | `LLM_PARSE_ERROR`     | 尝试提取 SQL；若失败，返回错误提示用户改写查询             |
| SQL 安全校验失败               | `VALIDATION_FAILED`   | 返回拒绝原因（违规类型），**不返回 SQL 原文**；`retryable: false` |
| SQL 超长（超阈值）             | `QUERY_TOO_LONG`      | 直接拒绝，提示用户简化查询；`retryable: false`             |
| SQL 执行超时                   | `EXECUTION_TIMEOUT`   | 取消查询，返回超时提示；`retryable: true`                  |
| SQL 执行失败（语法/语义错误）  | `EXECUTION_ERROR`     | 返回脱敏的错误摘要（不含表名/数据片段）和生成的 SQL        |
| 无法推断目标数据库             | `DB_AMBIGUOUS`        | 返回可用数据库名称列表，提示用户明确指定                   |
| 语义验证判定结果不匹配         | —                     | 自动重试（最多 2 次），最终结果仍附上验证说明              |
| 并发查询超过上限               | `RATE_LIMITED`        | 返回限流提示；`retryable: true`                            |

## 8. 约束与假设

### 8.1 约束

- 仅支持 PostgreSQL 数据库（不支持跨数据库查询）
- 仅支持只读查询操作（通过 AST 校验 + 只读事务双重保障）
- 依赖 DeepSeek API（需要网络连接与有效的 API Key）；LLM 输出视为不可信
- MCP 协议实现基于 Python MCP SDK
- 单语句限制：每次请求仅生成和执行一条 SQL 语句

### 8.2 假设

- 用户已正确配置 PostgreSQL 数据库连接信息，且连接用户具有所需表的 SELECT 权限及 `information_schema` / `pg_catalog` 的读取权限
- DeepSeek deepseek-chat 模型能够根据 schema 信息生成合理的 PostgreSQL SQL
- 数据库的 schema 在服务器运行期间相对稳定，不频繁变更（变更后可通过自然语言指令触发刷新）
- 默认部署场景为本地 stdio 模式，安全边界由 MCP 客户端保障

## 9. 未来扩展（不在本期范围内）

以下功能可能在后续版本中考虑，本期 **不** 纳入实现范围：

- 查询历史记录与缓存（相似问题 + schema hash 复用已生成的 SQL，减少 LLM 调用）
- 支持写操作（需要额外的审批/确认流程）
- 支持其他数据库（MySQL、SQLite 等）
- 支持其他 LLM 后端（Azure OpenAI、本地模型等）
- 查询结果可视化建议（如推荐使用柱状图/折线图展示）
- 多轮对话上下文（基于上一次查询继续追问）
- 向量化 schema 检索（使用 embedding 替代关键词匹配，提高相关 schema 子集召回率）
- 细粒度授权（按 schema / 表 / 列维度的访问控制）
- 分页/流式返回大结果集
