## 1. 概述

### 1.1 产品愿景

构建一个基于 Python 的 MCP (Model Context Protocol) 服务器，使 MCP 客户端（如 Cursor、Claude Desktop 等）能够通过自然语言描述来查询 PostgreSQL 数据库。服务器利用 OpenAI 大模型（gpt-5-mini）将自然语言转换为 SQL，并在执行前进行安全校验与结果验证，最终向用户返回生成的 SQL 语句或查询结果。

### 1.2 目标用户

- 使用 MCP 兼容客户端的开发者、数据分析师和数据库管理员
- 需要以自然语言方式快速查询 PostgreSQL 数据库的技术人员

### 1.3 核心价值

- 降低 SQL 编写门槛，通过自然语言即可完成数据库查询
- 自动感知数据库结构，无需用户手动描述 schema
- 内置安全防护，仅允许只读查询，防止数据篡改
- 智能结果验证，确保返回有意义的查询结果

## 2. 功能需求

### 2.1 数据库 Schema 自动发现与缓存

#### 2.1.1 启动时 Schema 采集

- **FR-2.1.1-01**: 服务器启动时，必须自动连接到所有已配置的 PostgreSQL 数据库实例
- **FR-2.1.1-02**: 对每个数据库，采集以下元数据并缓存到内存中：
  - **Tables（表）**: 表名、列名、列类型、列约束（NOT NULL / PRIMARY KEY / UNIQUE / DEFAULT 等）、列注释（如有）
  - **Views（视图）**: 视图名、列定义、底层查询定义
  - **Types（自定义类型）**: 枚举类型、复合类型等用户自定义数据类型的名称与定义
  - **Indexes（索引）**: 索引名、所属表、索引列、索引类型（B-tree / Hash / GIN / GiST 等）、是否唯一
  - **Schemas（模式）**: 所有非系统 schema 的名称（排除 `pg_catalog`、`information_schema` 等）
  - **Foreign Keys（外键关系）**: 外键约束名称、来源表/列、目标表/列，用于理解表间关联
  - **表/列注释（Comments）**: 通过 `pg_description` 获取表和列的注释信息，作为语义上下文提供给大模型
- **FR-2.1.1-03**: Schema 采集过程应记录日志，包括每个数据库的采集结果摘要（表数量、视图数量等）
- **FR-2.1.1-04**: 如果某个数据库连接失败，应记录错误日志并继续处理其他数据库，不应导致服务器整体启动失败

#### 2.1.2 Schema 缓存管理

- **FR-2.1.2-01**: 缓存的 Schema 信息应存储在内存中，支持按数据库名和 schema 名快速检索
- **FR-2.1.2-02**: 提供 Schema 缓存手动刷新机制（通过 MCP tool 调用触发），允许在数据库结构变更后更新缓存
- **FR-2.1.2-03**: 刷新操作应支持按指定数据库刷新，也支持全量刷新

### 2.2 自然语言转 SQL

#### 2.2.1 用户输入

- **FR-2.2.1-01**: 用户通过 MCP tool 调用提交自然语言查询请求，请求中包含：
  - **查询描述（必填）**: 自然语言描述的查询需求
  - **目标数据库（可选）**: 指定要查询的数据库名称；若未指定，服务器根据查询内容和已缓存的 schema 信息自动推断最合适的数据库
  - **返回模式（可选）**: 指定返回 SQL 语句还是查询结果，默认返回查询结果
    - `sql` — 仅返回生成的 SQL 语句
    - `result` — 执行 SQL 并返回查询结果
  - **结果行数限制（可选）**: 限制返回结果的最大行数，默认为 100 行

#### 2.2.2 SQL 生成

- **FR-2.2.2-01**: 服务器将用户的自然语言描述与目标数据库的 schema 信息组合为 prompt，调用 OpenAI gpt-5-mini 模型生成 SQL
- **FR-2.2.2-02**: 发送给大模型的上下文应包含：
  - 目标数据库的完整 schema 信息（表结构、关系、注释等）
  - 用户的自然语言查询描述
  - 明确的指令：只生成 SELECT/只读查询语句
  - 输出格式要求：仅输出可直接执行的 SQL 语句
- **FR-2.2.2-03**: 如果用户未指定目标数据库，系统应将用户查询与各数据库的 schema 摘要进行匹配，选择最相关的数据库（可以调用大模型辅助判断）

### 2.3 SQL 安全校验

- **FR-2.3-01**: 对生成的 SQL 进行语法解析，确保其为合法的 SQL 语句
- **FR-2.3-02**: 仅允许以下类型的 SQL 语句通过校验：
  - `SELECT` 查询语句（包括子查询）
  - `WITH ... SELECT`（CTE 查询）
  - `EXPLAIN` / `EXPLAIN ANALYZE`（查询计划分析，可选是否允许）
- **FR-2.3-03**: 必须拒绝以下类型的 SQL 语句：
  - 数据修改语句：`INSERT`、`UPDATE`、`DELETE`、`TRUNCATE`、`MERGE`
  - 结构修改语句：`CREATE`、`ALTER`、`DROP`
  - 权限管理语句：`GRANT`、`REVOKE`
  - 事务控制中的写操作
  - 调用可能产生副作用的函数（如 `pg_sleep` 以外的系统管理函数）
  - 包含 `INTO` 子句的 SELECT 语句（`SELECT ... INTO`）
- **FR-2.3-04**: 如果 SQL 未通过安全校验，应返回明确的错误信息说明拒绝原因，并且不执行该 SQL

### 2.4 SQL 执行与结果获取

- **FR-2.4-01**: 通过安全校验的 SQL 应在目标数据库上执行
- **FR-2.4-02**: 执行时必须使用只读事务（`SET TRANSACTION READ ONLY`），作为额外的安全保障
- **FR-2.4-03**: 执行必须设置语句超时（statement_timeout），防止长时间运行的查询拖垮数据库，默认超时时间可配置（建议默认 30 秒）
- **FR-2.4-04**: 获取查询结果时应遵循用户指定的行数限制（或默认限制），避免返回海量数据
- **FR-2.4-05**: 如果 SQL 执行失败，应捕获数据库错误信息并返回给用户，同时附上生成的 SQL 以便调试

### 2.5 结果语义验证

- **FR-2.5-01**: SQL 执行成功后，将以下信息提交给 OpenAI gpt-5-mini 进行语义验证：
  - 用户的原始自然语言查询
  - 生成的 SQL 语句
  - 查询结果的前 N 行样本数据（N 可配置，建议取前 5-10 行）
- **FR-2.5-02**: 大模型应评估并返回：
  - 结果是否与用户意图匹配（是/否/部分匹配）
  - 如果不匹配或部分匹配，给出原因说明
  - 如果有改进建议，提供优化后的 SQL（可选）
- **FR-2.5-03**: 如果语义验证认为结果不匹配且有优化建议，系统可选择自动重试（需限制最大重试次数，建议最多 2 次）
- **FR-2.5-04**: 最终返回给用户的响应应包含验证结果的摘要信息（如匹配度评价）
- **FR-2.5-05**: 语义验证为可选步骤，用户可通过参数控制是否启用，默认启用

### 2.6 结果返回

- **FR-2.6-01**: 根据用户指定的返回模式返回相应内容：
  - **`sql` 模式**: 返回生成的 SQL 语句，以及语义验证摘要（如果启用了验证）
  - **`result` 模式**: 返回查询结果数据，同时附上生成的 SQL 和语义验证摘要
- **FR-2.6-02**: 查询结果应以结构化格式返回，包含：
  - 列名列表
  - 数据行（列表形式）
  - 总行数
  - 是否被截断（即实际结果是否超过行数限制）
- **FR-2.6-03**: 对于大结果集，应在返回中明确标注结果已截断，并提示实际匹配的行数（如果可用）

## 3. MCP 接口定义

### 3.1 MCP Tools

服务器应暴露以下 MCP Tools：

#### 3.1.1 `query` — 自然语言查询

主功能工具，接受自然语言并返回 SQL 或查询结果。

**输入参数:**

| 参数名          | 类型     | 必填 | 默认值     | 说明                                                  |
| --------------- | -------- | ---- | ---------- | ----------------------------------------------------- |
| `question`      | string   | 是   | —          | 自然语言描述的查询需求                                |
| `database`      | string   | 否   | 自动推断   | 目标数据库名称                                        |
| `return_mode`   | string   | 否   | `"result"` | 返回模式：`"sql"` 或 `"result"`                       |
| `max_rows`      | integer  | 否   | `100`      | 返回结果的最大行数                                    |
| `verify_result` | boolean  | 否   | `true`     | 是否启用语义验证                                      |

**输出:**

| 字段名          | 类型     | 说明                                                  |
| --------------- | -------- | ----------------------------------------------------- |
| `sql`           | string   | 生成的 SQL 语句                                       |
| `database`      | string   | 实际查询的数据库名称                                  |
| `result`        | object   | 查询结果（仅 `result` 模式）；含 `columns`、`rows`、`row_count`、`truncated` 字段 |
| `verification`  | object   | 语义验证结果；含 `match`、`explanation` 字段          |
| `error`         | string   | 错误信息（如有）                                      |

#### 3.1.2 `list_databases` — 列出可用数据库

返回服务器已连接且缓存了 schema 的所有数据库列表。

**输入参数:** 无

**输出:**

| 字段名       | 类型   | 说明                                      |
| ------------ | ------ | ----------------------------------------- |
| `databases`  | array  | 数据库信息列表；每项含 `name`、`schemas`（schema 列表）、`table_count`、`view_count` |

#### 3.1.3 `describe_table` — 查看表结构

返回指定表的详细结构信息。

**输入参数:**

| 参数名      | 类型   | 必填 | 说明         |
| ----------- | ------ | ---- | ------------ |
| `database`  | string | 是   | 数据库名称   |
| `table`     | string | 是   | 表名（可含 schema 前缀，如 `public.users`） |

**输出:** 表的列定义、索引、外键等详细信息。

#### 3.1.4 `refresh_schema` — 刷新 Schema 缓存

触发 schema 缓存刷新。

**输入参数:**

| 参数名      | 类型   | 必填 | 默认值 | 说明                                  |
| ----------- | ------ | ---- | ------ | ------------------------------------- |
| `database`  | string | 否   | 全部   | 指定要刷新的数据库名称；省略则刷新全部 |

**输出:** 刷新结果摘要。

### 3.2 MCP Resources

服务器应暴露以下 MCP Resources：

| Resource URI                          | 说明                              |
| ------------------------------------- | --------------------------------- |
| `pg://databases`                      | 所有已连接数据库的概览            |
| `pg://{database}/schema`              | 指定数据库的完整 schema 信息      |
| `pg://{database}/tables`              | 指定数据库的所有表列表及摘要      |
| `pg://{database}/tables/{table_name}` | 指定表的详细结构                  |

## 4. 非功能需求

### 4.1 安全性

- **NFR-4.1-01**: 数据库连接凭据不得硬编码在代码中，必须通过环境变量或配置文件提供
- **NFR-4.1-02**: OpenAI API Key 必须通过环境变量提供
- **NFR-4.1-03**: 所有数据库操作必须使用只读连接/事务，从数据库层面防止意外写入
- **NFR-4.1-04**: SQL 安全校验必须在代码层面进行（不仅依赖数据库权限），提供纵深防御
- **NFR-4.1-05**: 敏感信息（如连接字符串、API Key）不得出现在日志中

### 4.2 性能

- **NFR-4.2-01**: Schema 缓存应使服务器在接收到查询请求时不再需要实时查询数据库元数据
- **NFR-4.2-02**: 单次查询的端到端响应时间目标：< 15 秒（不含数据库查询执行时间）
- **NFR-4.2-03**: SQL 执行超时可配置，默认 30 秒

### 4.3 可靠性

- **NFR-4.3-01**: 单个数据库连接失败不应影响服务器对其他数据库的服务能力
- **NFR-4.3-02**: OpenAI API 调用失败时应返回明确的错误信息，建议用户重试
- **NFR-4.3-03**: 数据库连接应具备断线重连机制

### 4.4 可观测性

- **NFR-4.4-01**: 记录结构化日志，包含每次查询的完整生命周期：接收请求 → SQL 生成 → 安全校验 → SQL 执行 → 语义验证 → 返回结果
- **NFR-4.4-02**: 日志级别应可配置（DEBUG / INFO / WARNING / ERROR）

### 4.5 可配置性

- **NFR-4.5-01**: 以下参数应通过环境变量或配置文件进行配置：
  - PostgreSQL 数据库连接信息（支持多个数据库）
  - OpenAI API Key 及模型名称
  - SQL 执行超时时间
  - 默认结果行数限制
  - 语义验证的样本行数
  - 日志级别
  - 语义验证是否默认启用

## 5. 配置方式

### 5.1 数据库连接配置

通过环境变量配置一个或多个 PostgreSQL 数据库连接：

```
PG_MCP_DATABASES=db1,db2,db3
PG_MCP_DB1_HOST=localhost
PG_MCP_DB1_PORT=5432
PG_MCP_DB1_DATABASE=mydb1
PG_MCP_DB1_USER=readonly_user
PG_MCP_DB1_PASSWORD=secret
PG_MCP_DB1_SSLMODE=prefer
```

或使用连接字符串：

```
PG_MCP_DB1_URL=postgresql://readonly_user:secret@localhost:5432/mydb1?sslmode=prefer
```

### 5.2 OpenAI 配置

```
OPENAI_API_KEY=sk-...
OPENAI_MODEL=gpt-5-mini
```

### 5.3 服务器运行参数

```
PG_MCP_STATEMENT_TIMEOUT=30
PG_MCP_DEFAULT_MAX_ROWS=100
PG_MCP_VERIFY_SAMPLE_ROWS=5
PG_MCP_LOG_LEVEL=INFO
PG_MCP_ENABLE_VERIFICATION=true
```

## 7. 错误场景与处理

| 场景                           | 处理方式                                                   |
| ------------------------------ | ---------------------------------------------------------- |
| 数据库连接失败（启动时）       | 记录错误日志，跳过该数据库，继续启动                       |
| 数据库连接失败（查询时）       | 返回错误信息，提示用户该数据库暂不可用                     |
| OpenAI API 调用失败            | 返回错误信息，提示用户稍后重试                             |
| OpenAI 返回非法 SQL            | 尝试提取 SQL 部分；若仍无法解析，返回错误信息              |
| SQL 安全校验失败               | 返回错误信息，说明检测到非只读操作，拒绝执行               |
| SQL 执行超时                   | 取消查询，返回超时错误信息                                 |
| SQL 执行失败（语法/语义错误）  | 返回数据库错误信息及生成的 SQL，供用户调试                 |
| 无法推断目标数据库             | 返回可用数据库列表，提示用户明确指定                       |
| 语义验证判定结果不匹配         | 尝试重新生成 SQL（最多重试 2 次），最终结果仍附上验证说明  |

## 8. 约束与假设

### 8.1 约束

- 仅支持 PostgreSQL 数据库
- 仅支持只读查询操作
- 依赖 OpenAI API（需要网络连接与有效的 API Key）
- MCP 协议实现基于 Python MCP SDK

### 8.2 假设

- 用户已正确配置 PostgreSQL 数据库连接信息，且连接用户具有所需表的 SELECT 权限
- OpenAI gpt-5-mini 模型能够根据 schema 信息生成合理的 PostgreSQL SQL
- 数据库的 schema 在服务器运行期间相对稳定，不频繁变更

## 9. 未来扩展（不在本期范围内）

以下功能可能在后续版本中考虑，本期 **不** 纳入实现范围：

- 查询历史记录与缓存（相似问题复用已生成的 SQL）
- 支持写操作（需要额外的审批/确认流程）
- 支持其他数据库（MySQL、SQLite 等）
- 支持其他 LLM 后端（Azure OpenAI、本地模型等）
- 查询结果可视化建议（如推荐使用柱状图/折线图展示）
- 多轮对话上下文（基于上一次查询继续追问）

## 10. 术语表

| 术语         | 说明                                                                 |
| ------------ | -------------------------------------------------------------------- |
| MCP          | Model Context Protocol，由 Anthropic 提出的模型上下文协议            |
| MCP Tool     | MCP 协议中服务器暴露给客户端可调用的功能接口                         |
| MCP Resource | MCP 协议中服务器暴露给客户端可读取的数据资源                         |
| Schema       | 数据库中的命名空间/模式，也泛指数据库的结构定义（表、列、类型等）    |
| CTE          | Common Table Expression，公共表表达式（WITH 语句）                   |
| gpt-5-mini   | OpenAI 的大语言模型，用于自然语言到 SQL 的转换及语义验证             |
